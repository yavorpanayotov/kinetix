# Diagnostic: Gateway 500-on-404/503 Exception Path Audit

**Date:** 2026-05-26
**Issue:** kx-9v7
**Test pinning behaviour:** `UpstreamExceptionPathDiagnosticTest` (10 tests, all green)

## Executive summary

The original audit hypothesised that 5 gateway endpoints returned HTTP 500 instead of
404/503 when the upstream (risk-orchestrator) was absent or returned an error.  The
architect-review correction was correct: `Application.kt:170` installs an explicit
`exception<UpstreamErrorException>` handler that reflects `cause.statusCode` back to the
caller.  All 5 endpoints already return the correct status code for upstream-404.
No endpoint hits the generic `Throwable` catch-all for the error cases exercised here.

## Global error-handling architecture

`Application.kt` StatusPages handlers (in precedence order):

| Handler | Status emitted |
|---|---|
| `ServiceUnavailableException` | 503 + `Retry-After` header |
| `GatewayTimeoutException` | 504 |
| `UpstreamErrorException` | reflects `cause.statusCode` (e.g. 404, 500) |
| `JsonConvertException` | 400 `invalid_request_body` |
| `SerializationException` | 400 `invalid_request_body` |
| `BadRequestException` | 400 `bad_request` / `invalid_request_body` |
| `IllegalArgumentException` | 400 `bad_request` |
| `Throwable` (catch-all) | 500 `internal_error` |

`HttpRiskServiceClient.handleErrorResponse()` maps upstream non-success responses to:
- 503 upstream → `ServiceUnavailableException`
- 504 upstream → `GatewayTimeoutException`
- any other status → `UpstreamErrorException(statusCode, message)`

## Per-endpoint findings

### 1. GET /api/v1/risk/krd/{bookId}

**Client method:** `getKeyRateDurations(bookId)`
**Upstream-404 path:** client checks `if (response.status == HttpStatusCode.NotFound) return null`; route handler maps null to `call.respond(HttpStatusCode.NotFound)`.
**Exception thrown:** none — null path, no exception.
**Gateway response (upstream-404):** 404
**Gateway response (upstream-500):** `UpstreamErrorException(500)` thrown by `handleErrorResponse`; caught by Application.kt:170; reflected as 500 with `{"error":"upstream_error",...}`.
**Fix needed in:** nothing for 404. Behaviour is already correct.

**Upstream-404 path:** identical null-return pattern to endpoint 1.
**Exception thrown:** none — null path.
**Gateway response (upstream-404):** 404

**Fix needed in:** nothing for 404. Behaviour is already correct.

### 4. GET /api/v1/risk/jobs/{bookId}/chart

**Client method:** `getChartData(bookId, from, to)` — returns non-nullable `ChartDataSummary`.
**Upstream-404 path:** no `NotFound → null` short-circuit. Instead:
  `if (!response.status.isSuccess()) handleErrorResponse(response)` is reached;
  `handleErrorResponse` throws `UpstreamErrorException(404, "Not Found")`;
  Application.kt:170 catches it and reflects 404.
  `response.body<ChartDataClientDto>()` is never reached, so no deserialization NPE can occur.
**Exception thrown:** `UpstreamErrorException(404)` — caught by the explicit handler.
**Gateway response (upstream-404):** 404 with `{"error":"upstream_error","message":"Not Found"}`
**Gateway response (upstream-500):** `UpstreamErrorException(500)` reflected as 500.
**Hypothesis disproved:** The NPE-during-deserialization hypothesis was incorrect.
  `handleErrorResponse()` throws before `response.body()` is ever called.
**Residual risk (different scenario):** If upstream returns HTTP 200 with a malformed/empty
  body, `response.body<ChartDataClientDto>()` would throw `SerializationException`, which IS
  now caught by Application.kt:189 and mapped to 400 `invalid_request_body` — not 500.
**Fix needed in:** nothing for 404 or 500. The non-nullable return type creates no 500 risk
  for these error cases, only a semantic difference (returns 404 with an `upstream_error` body
  vs. a plain empty 404). A future improvement could add a `NotFound → null` short-circuit and
  return a clean 404 with no body, but this is cosmetic, not a bug.

## Where the "500 instead of 404" bug actually lives

None of the 5 endpoints produce gateway-500 when the upstream returns 404.  The original
audit's premise — that `UpstreamErrorException` propagates uncaught to the `Throwable`
catch-all — was incorrect.  The explicit handler at Application.kt:170 has been in place
since before this audit.

The only way a gateway-500 can arise from these routes is if the upstream returns a
non-4xx/non-5xx status that bypasses all checks, which is not possible in HTTP/1.1, or if
there is a bug in the Ktor runtime itself.

If production logs showed actual 500s from these endpoints, the cause is likely one of:

- A network-level error (connection refused, timeout) that propagates as a Ktor
  `java.net.ConnectException` or `io.ktor.client.plugins.HttpRequestTimeoutException`,
  neither of which is caught by the explicit handlers and both of which fall to the
  `Throwable` catch-all → 500.  Adding `ConnectException` and `HttpRequestTimeoutException`
  handlers (mapped to 503 `ServiceUnavailableException`) would fix this class of error.
- A misconfiguration of the upstream URL (env var wrong in a given deployment) causing the
  Ktor CIO client to fail at the TCP level, same root cause as above.

## Recommended follow-up (out of scope for this diagnostic)

- Add a `ConnectException` / `HttpRequestTimeoutException` → 503 handler in StatusPages.
  This is the single highest-impact change to eliminate gateway-500 on upstream-unavailable.
- For endpoint 4 (`getChartData`), consider adding `NotFound → null` handling and making the
  return type nullable so the route can return a clean 404 with no body, consistent with the
  other endpoints.
