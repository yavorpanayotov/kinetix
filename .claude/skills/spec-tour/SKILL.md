# Spec Tour

Produce a **guided walk-through** of one Allium specification — from the spec, through `/propagate`-generated tests, into the implementation, and finally through the divergence-report loop. This is the document that makes Kinetix's spec-driven development story tangible to a reader who has never seen Allium.

The audience is an engineer, technical evaluator, or interview panel asking "what does spec-driven AI development actually look like?".

## Step 1 — Resolve the spec

Argument names a spec (without `.allium`). Resolve to `specs/<name>.allium`. If the user gave no argument, list available specs (`ls specs/*.allium`) and ask, or default to the spec with the most recent divergence report under `specs/divergences/`.

Read `specs/README.md` for the spec-system conventions.

## Step 2 — Section: The spec itself

- Quote the first 30–60 lines of the spec as a code block.
- Explain in plain English what the spec promises: entities, rules, triggers, surfaces, contracts.
- Highlight 2–3 of the most interesting clauses (rules with non-trivial guards, triggers with debouncing, surfaces with invariants).

Pair the prose with line-number references back into the spec file so the reader can dig deeper.

## Step 3 — Section: Generated tests

Locate tests generated by `/propagate` for this spec. Heuristics:
- Grep test files in the relevant service for "Generated from spec" markers or the spec name.
- Look at `propagate` output conventions documented in the propagate skill.

For each meaningful test family (state-machine, property-based, contract):
- Show one representative test as a code block.
- Explain which spec clause it validates.
- Note the test type (Kotest FunSpec, pytest property, Hypothesis, Playwright).

If no propagated tests are found for this spec, say so and recommend running `/propagate <spec>`.

## Step 4 — Section: Implementation

Locate the code that implements the spec. Heuristics:
- Service whose name matches the spec topic (e.g. `limits.allium` → `position-service` limits routes/repositories).
- ADRs that cite the spec.
- Grep for entity names from the spec across services.

For each implementation file:
- Cite the file path and a short excerpt (5–15 lines).
- Map the excerpt back to the spec clause it satisfies.

Aim for 3–5 implementation references — enough to show how a spec lands in code, not exhaustive.

## Step 5 — Section: The divergence loop

If `specs/divergences/<spec>*.md` exists, walk through the most recent divergence report:
- What `/weed` found.
- How it was resolved (spec updated, code updated, both).
- The commits or PRs that closed the gap.

If no divergence report exists, demonstrate the loop hypothetically: "if `/weed` found `<concrete example clause>` was not implemented, the flow would be — update spec, run `/propagate`, implement, re-run `/weed`."

## Step 6 — Section: Why this matters

Half a page on the meta point:
- Specs are versioned, AI-readable behavioural contracts.
- Tests are generated, not invented — so coverage tracks the spec, not developer intuition.
- Divergences are surfaced automatically, so drift is measurable.
- The same agent can author the spec, propagate tests, and implement code under TDD — a single coherent loop instead of three disconnected disciplines.

Cite ADRs 0031–0034 if relevant (these tie ADR decisions to spec lines).

## Step 7 — Output

Write to `docs/spec-tour/<spec>.md`. Create `docs/spec-tour/` if needed. Print the file path.

## Reminders

- This is a tour, not a reference. Keep prose moving; let the code blocks do the heavy lifting.
- Cite every code block by file path and line range. No invented snippets.
- Pair with `/case-study` — a case study explains *what* Kinetix is; a spec tour explains *how it is built*. Together they're a strong pair for a JUXT-style prospect deck.
- Link out to the Allium project (`https://github.com/juxt/allium`) once, in the "Why this matters" section.
